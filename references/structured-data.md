# Structured Data (Rich Results)

Ground truth from Google's structured-data docs: which feature maps to which schema.org `@type`, the top-level required properties, which features are gated (limited-access / invite-only / beta / region-locked), and the binding content policies. Every factual claim is an exact quote + a repo-relative source path. `UNKNOWN` appears only where the source page genuinely does not state a value.

The four features that a prior extraction truncated ‚Äî `local-business`, `video`, `profile-page`, `qapage` ‚Äî are now resolved from their actual "Required properties" tables (see the gating table below).

## Per-feature binding constraints (verbatim, from each page's own guidelines)

Exact quotes ‚Äî these are the "do NOT tell a client to just add the required props" gotchas for the 14 newly added features:

- **Course ‚Äî at least three courses**: "You must mark up at least three courses. The courses can be on separate detail pages, or in an all-in-one page." ‚Äî `appearance/structured-data/course.md`. Each course needs valid `name` and `provider`; promotional titles ("Best school in the world", "Learn ukulele - only $30!") are invalid.
- **Dataset ‚Äî description length bounds**: "The summary must be between 50 and 5000 characters long. The summary may include Markdown syntax." (`description` property) ‚Äî `appearance/structured-data/dataset.md`.
- **Discussion forum ‚Äî usage restriction**: "Only use `DiscussionForumPosting` markup to describe a user-generated post on a website. Don't use this markup for content that's primarily authored by the publishers of the website or their agents." Invalid: "An article or blog written directly by an agent for the website (even with comments)" and "User reviews about a product". ‚Äî `appearance/structured-data/discussion-forum.md`.
- **Employer aggregate rating ‚Äî user-ratings requirement**: "Users must be able to post their own ratings on your site and your site must host those user ratings." and "The number of ratings must reflect actual ratings that users provide." ‚Äî `appearance/structured-data/employer-rating.md`. Required table also notes: "At least one of `ratingCount` or `reviewCount` is required."
- **Fact check ‚Äî one `ClaimReview` per page**: "To be eligible for the single fact check rich result, a page must only have one `ClaimReview` element. If you add multiple `ClaimReview` elements per page, the page won't be eligible for the single fact check rich result." and "A specific `ClaimReview` must only be on one page on your site." ‚Äî `appearance/structured-data/factcheck.md`.
- **Image metadata ‚Äî `license` property requirement**: "If you're using structured data to specify an image, you must include the `license` property for your image to be eligible to be shown with the Licensable badge." (Top-level required is `contentUrl` + one of `creator`/`creditText`/`copyrightNotice`/`license`.) ‚Äî `appearance/structured-data/image-license-metadata.md`.
- **Loyalty program ‚Äî at least one tier**: "`hasTiers` ‚Ä¶ Defines a tier under a loyalty program. A loyalty program must have at least one tier." ‚Äî `appearance/structured-data/loyalty-program.md`.
- **Math solvers ‚Äî login/paywall rule**: "We don't allow math solvers that are entirely hidden behind a login or paywall. Once users navigate from the feature on Google to your site, the solution and a step-by-step walkthrough for their initial problem must be accessible to them. Additional content can be behind a login or paywall." ‚Äî `appearance/structured-data/math-solvers.md`.
- **Movie ‚Äî carousel mobile-only**: "The movie carousel is only available on mobile devices." ‚Äî `appearance/structured-data/movie.md`.
- **Product variants ‚Äî unique variant ID**: "Each variant must have a unique ID in its corresponding structured data markup (for example, using the `sku` or `gtin` properties)." and "Each product group must have a unique ID ‚Ä¶ specified with the `inProductGroupWithID` property in variant `Product` properties or the `productGroupID` property in the `ProductGroup` property." ‚Äî `appearance/structured-data/product-variants.md`.
- **Return policy ‚Äî `MerchantReturnFiniteReturnWindow` forces `merchantReturnDays`**: "If you use `MerchantReturnFiniteReturnWindow`, then the `merchantReturnDays` property is required." (`merchantReturnDays` is otherwise listed as recommended.) ‚Äî `appearance/structured-data/return-policy.md`.
- **Shipping policy ‚Äî `ShippingService` required props**: "`shippingConditions` ‚Ä¶ Specify the shipping cost and/or delivery times that apply for a particular set of conditions." is the only required `ShippingService` property; nested `ShippingConditions` has no required properties (all recommended), and nested `DefinedRegion` requires `addressCountry`. ‚Äî `appearance/structured-data/shipping-policy.md`.
- **Software app ‚Äî `VideoGame`-only exclusion**: "Google doesn't show a rich result for Software Apps that only have the `VideoGame` type. To make sure that your Software App is eligible for display as a rich result, co-type the `VideoGame` type with another type." (Required: `name`, `offers.price`, and one of `aggregateRating`/`review`.) ‚Äî `appearance/structured-data/software-app.md`.
- **Search gallery** ‚Äî reference index only: "Structured data markup that Google Search supports" ‚Äî the master list of every rich-result feature; it declares no `@type` and no required properties of its own. ‚Äî `appearance/structured-data/search-gallery.md`.

## Hard requirements

### sd-policies ‚Äî binding across ALL features (visibility, no fabrication, no markup for off-page content)

Exact quotes from `docs/search/docs/appearance/structured-data/sd-policies.md` unless noted:

- Visibility: "Don't mark up content that is not visible to readers of the page. For example, if the JSON-LD markup describes a performer, the HTML body must describe that same performer." ‚Äî `appearance/structured-data/sd-policies.md`
- No markup for content not on the page (intro): "Don't create blank or empty pages just to hold structured data, and don't add structured data about information that is not visible to the user, even if the information is accurate." ‚Äî `appearance/structured-data/intro-structured-data.md`
- No fake/misleading content: "Don't mark up irrelevant or misleading content, such as fake reviews or content unrelated to the focus of a page." ‚Äî `appearance/structured-data/sd-policies.md`
- No deception/impersonation: "Don't use structured data to deceive or mislead users. Don't impersonate any person or organization, or misrepresent your ownership, affiliation, or primary purpose." ‚Äî `appearance/structured-data/sd-policies.md`
- Original content: "Provide original content that you or your users have generated." ‚Äî `appearance/structured-data/sd-policies.md`
- Up to date: "Provide up-to-date information. We won't show a rich result for time-sensitive content that is no longer relevant." ‚Äî `appearance/structured-data/sd-policies.md`
- Fabricated ratings ‚Üí manual action: "reviews or ratings not by actual users may result in manual action" ‚Äî `appearance/structured-data/sd-policies.md`
- Relevance: "Your structured data must be a true representation of the page content." ‚Äî `appearance/structured-data/sd-policies.md`
- Completeness: "Specify all required properties listed in the documentation for your specific rich result type. Items that are missing required properties are not eligible for rich results." ‚Äî `appearance/structured-data/sd-policies.md`
- Access: "Don't block your structured data pages to Googlebot using robots.txt, `noindex`, or any other access control methods." ‚Äî `appearance/structured-data/sd-policies.md`

Consequence / non-guarantee:

- "A structured data manual action means that a page loses eligibility for appearance as a rich result; it doesn't affect how the page ranks in Google web search." ‚Äî `appearance/structured-data/sd-policies.md`
- "Google does not guarantee that your structured data will show up in search results, even if your page is marked up correctly according to the Rich Results Test." ‚Äî `appearance/structured-data/sd-policies.md`

### Review-snippet rules ‚Äî self-serving restriction + required AggregateRating/Review props

Provenance / self-serving ‚Äî exact quotes from `docs/search/docs/appearance/structured-data/review-snippet.md`:

- Self-serving reviews are ineligible: "If the entity that's being reviewed controls the reviews about itself, their pages that use `LocalBusiness` or any other type of `Organization` structured data are ineligible for star review feature." ‚Äî `appearance/structured-data/review-snippet.md`
- "Ratings must be sourced directly from users." ‚Äî `appearance/structured-data/review-snippet.md`
- "Don't rely on human editors to create, curate, or compile ratings information for local businesses." ‚Äî `appearance/structured-data/review-snippet.md`
- "Don't aggregate reviews or ratings from other websites." ‚Äî `appearance/structured-data/review-snippet.md`
- Visibility: "Make sure the review content you mark up are readily available to users from the marked-up page. It must be immediately obvious to users that the page has review content." ‚Äî `appearance/structured-data/review-snippet.md`
- "If you include multiple individual reviews, also include an aggregate rating of the individual reviews." ‚Äî `appearance/structured-data/review-snippet.md`
- "Provide review information about a specific item, not about a category or a list of items." ‚Äî `appearance/structured-data/review-snippet.md`
- Valid author name: "The reviewer's name must be a valid name. For example, \"50% off until Saturday\" is not a valid name for a reviewer." ‚Äî `appearance/structured-data/review-snippet.md`
- `LocalBusiness`/`Organization` reviews eligible only "for sites that capture reviews about other local businesses" / "for sites that capture reviews about other organizations". ‚Äî `appearance/structured-data/review-snippet.md`

`AggregateRating` required properties (exact) ‚Äî `docs/search/docs/appearance/structured-data/review-snippet.md`:

- `itemReviewed` (if not a Nested Aggregate Rating) ‚Äî "The item that is being rated." (omit if nested via `aggregateRating`).
- `itemReviewed.name` or parent `name` ‚Äî "The name of the item that is being reviewed."
- `ratingCount` ‚Äî "The total number of ratings for the item on your site. At least one of `ratingCount` or `reviewCount` is required."
- `reviewCount` ‚Äî "Specifies the number of people who provided a review with or without an accompanying rating. At least one of `ratingCount` or `reviewCount` is required."
- `ratingValue` ‚Äî "The average rating for the item's quality using a numerical rating..."
- Recommended: `bestRating` ("If `bestRating` is omitted, 5 is assumed."), `worstRating` ("If `worstRating` is omitted, 1 is assumed.").

`Review` required properties (exact): `author`, `itemReviewed` (if not a Nested Review), `itemReviewed.name`/parent `name`, `reviewRating`, `reviewRating.ratingValue`. Author length rule: "This field must be shorter than 100 characters. If it's longer than 100 characters, your page won't be eligible for an author-based review snippet." ‚Äî `appearance/structured-data/review-snippet.md`

Formatting rule: "For decimal numbers, use a dot instead of a comma to specify the value (for example `4.4` instead of `4,4`)." ‚Äî `appearance/structured-data/review-snippet.md`

## Recommendations

- Prefer JSON-LD: "In general, Google recommends using JSON-LD for structured data if your site's setup allows it, as it's the easiest solution for website owners to implement and maintain at scale." ‚Äî `appearance/structured-data/intro-structured-data.md`
- Google Search Central docs win over schema.org: "you should rely on the Google Search Central documentation as definitive for Google Search behavior, rather than the schema.org documentation." ‚Äî `appearance/structured-data/intro-structured-data.md`
- Quality over quantity: "it is more important to supply fewer but complete and accurate recommended properties rather than trying to provide every possible recommended property with less complete, badly-formed, or inaccurate data." ‚Äî `appearance/structured-data/intro-structured-data.md`
- Specificity: "Try to use the most specific applicable type and property names defined by schema.org for your markup." ‚Äî `appearance/structured-data/sd-policies.md`
- Duplicates: "If you have duplicate pages for the same content, we recommend placing the same structured data on all page duplicates, not just on the canonical page." ‚Äî `appearance/structured-data/sd-policies.md`
- Images must be crawlable: "All image URLs specified in structured data must be crawlable and indexable." ‚Äî `appearance/structured-data/sd-policies.md`
- Monitor after deploy: "Be sure to check your structured data using the Rich Results Test during development, and the Rich result status reports after deployment." ‚Äî `appearance/structured-data/intro-structured-data.md`
